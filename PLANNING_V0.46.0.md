# Planning v0.46.0: WGSL Transpiler & Auto-Module Generation

## 🔮 What's Next (v0.46.0)

### Planned Features

#### 1. **Windjammer→WGSL Transpiler**
GPU shader compilation from Windjammer with type-safe CPU/GPU interop.

**Goals:**
- Compile Windjammer code to WGSL (WebGPU Shading Language)
- Type-safe struct layout between CPU and GPU
- Automatic alignment and padding (match WGSL requirements)
- Compile-time dispatch validation (bind groups, entry points)

**Benefits:**
- Write GPU shaders in Windjammer syntax
- Automatic memory layout validation
- Prevent runtime GPU errors at compile time
- Unified CPU/GPU type system

**Design Considerations:**
- WGSL has strict alignment rules (e.g., vec3 aligns to 16 bytes)
- CPU struct layout vs GPU struct layout (add padding automatically)
- Shader entry point validation (vertex, fragment, compute)
- Bind group layout generation
- Uniform/storage buffer type checking

**Implementation Phases:**
1. **Phase 1: Basic Type Mapping**
   - Map Windjammer types to WGSL types
   - Handle primitives (f32, i32, u32, bool)
   - Handle vectors (vec2, vec3, vec4)
   - Handle matrices (mat2x2, mat3x3, mat4x4)

2. **Phase 2: Struct Layout & Alignment**
   - Analyze struct layout for GPU compatibility
   - Automatic padding insertion
   - Alignment validation
   - Size calculation

3. **Phase 3: Entry Point Generation**
   - Vertex shader entry points
   - Fragment shader entry points
   - Compute shader entry points
   - Stage validation

4. **Phase 4: Bind Group Generation**
   - Uniform buffers
   - Storage buffers
   - Textures and samplers
   - Bind group layout descriptors

**Example:**
```windjammer
// Windjammer shader code
@vertex
pub fn vertex_main(
    @location(0) position: vec3<f32>,
    @location(1) uv: vec2<f32>,
) -> @builtin(position) vec4<f32> {
    vec4(position.x, position.y, position.z, 1.0)
}

@fragment
pub fn fragment_main(
    @location(0) uv: vec2<f32>,
) -> @location(0) vec4<f32> {
    vec4(uv.x, uv.y, 0.0, 1.0)
}
```

**Compiles to WGSL:**
```wgsl
@vertex
fn vertex_main(
    @location(0) position: vec3<f32>,
    @location(1) uv: vec2<f32>,
) -> @builtin(position) vec4<f32> {
    return vec4<f32>(position.x, position.y, position.z, 1.0);
}

@fragment
fn fragment_main(
    @location(0) uv: vec2<f32>,
) -> @location(0) vec4<f32> {
    return vec4<f32>(uv.x, uv.y, 0.0, 1.0);
}
```

#### 2. **Auto-Module Generation**
Eliminate manual `#[path]` stubs and scale to 1000+ modules.

**Goals:**
- Automatically generate `windjammer_modules.rs`
- Discover all `.wj` files in the project
- Generate correct Rust module declarations
- Handle nested directory structures
- Scale to large projects (1000+ modules)

**Benefits:**
- No manual module registration
- Add new `.wj` files without touching Rust code
- Automatic module tree generation
- Support for deeply nested module hierarchies

**Design Considerations:**
- Module naming conventions (snake_case from file paths)
- Visibility rules (pub vs private modules)
- Re-exports (which modules to expose at root)
- Circular dependency detection
- Module path resolution

**Implementation Phases:**
1. **Phase 1: Module Discovery**
   - Scan project for all `.wj` files
   - Build module tree from file structure
   - Generate module paths

2. **Phase 2: Code Generation**
   - Generate `mod` declarations
   - Generate `pub use` statements
   - Handle nested modules
   - Generate `#[path]` annotations

3. **Phase 3: Integration**
   - Hook into `wj build` workflow
   - Regenerate on file changes
   - Cache for incremental builds

4. **Phase 4: Optimization**
   - Parallel module discovery
   - Incremental updates
   - Handle 1000+ modules efficiently

**Example:**

**Project Structure:**
```
src_wj/
  ├── math/
  │   ├── vector.wj
  │   ├── matrix.wj
  │   └── quaternion.wj
  ├── physics/
  │   ├── body.wj
  │   ├── collider.wj
  │   └── constraint.wj
  └── renderer/
      ├── camera.wj
      ├── mesh.wj
      └── material.wj
```

**Generated `windjammer_modules.rs`:**
```rust
// Auto-generated by Windjammer compiler
// DO NOT EDIT - This file is regenerated on build

#[path = "math/vector.rs"]
pub mod vector;

#[path = "math/matrix.rs"]
pub mod matrix;

#[path = "math/quaternion.rs"]
pub mod quaternion;

#[path = "physics/body.rs"]
pub mod body;

#[path = "physics/collider.rs"]
pub mod collider;

#[path = "physics/constraint.rs"]
pub mod constraint;

#[path = "renderer/camera.rs"]
pub mod camera;

#[path = "renderer/mesh.rs"]
pub mod mesh;

#[path = "renderer/material.rs"]
pub mod material;

// Re-exports for convenience
pub mod math {
    pub use super::vector::*;
    pub use super::matrix::*;
    pub use super::quaternion::*;
}

pub mod physics {
    pub use super::body::*;
    pub use super::collider::*;
    pub use super::constraint::*;
}

pub mod renderer {
    pub use super::camera::*;
    pub use super::mesh::*;
    pub use super::material::*;
}
```

**Usage:**
```rust
// In lib.rs or main.rs
include!("windjammer_modules.rs");

// Now all modules are available:
use math::vector::Vec3;
use physics::body::RigidBody;
use renderer::camera::Camera;
```
