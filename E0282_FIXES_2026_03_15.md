# E0282 Type Inference Fixes - 2026-03-15

## Summary

Two critical fixes to reduce E0282 "type annotations needed" errors by exploiting available context.

## Fix 1: MethodCall Handling for Vec::new() (ROOT CAUSE)

**Problem:** The Windjammer parser produces `Expression::MethodCall` for `Vec::new()`, not `Expression::Call`. The E0282 inference logic only matched `Expression::Call` with `FieldAccess`, so inference never ran.

**Evidence:** Generated code had `let mut result = Vec::new();` without type annotation even when:
- `result` was returned and fn returned `Vec<&Achievement>`
- `result.push(ach)` appeared in a for-loop over `map.values()`

**Fix:** Added `Expression::MethodCall` handling in `statement_generation.rs` (before the `Expression::Call` branch) with identical inference logic for:
- `Vec::new()` → infer from `.push()` or return type
- `HashMap::new()` → infer from `.insert()`
- `HashSet::new()` → infer from `.insert()`

**Files:** `windjammer/src/codegen/rust/statement_generation.rs`

**Impact:** Fixes 15+ E0282 errors in achievement/manager, dialogue/manager, inventory, etc.

## Fix 2: Return Vec::new() Turbofish from Function Return Type

**Problem:** `return Vec::new()` when fn returns `Vec<u32>` generated `return Vec::new()` without turbofish → E0282.

**Fix:** In `expression_generation.rs` MethodCall handler, when `type_args` is None and we have `Vec::new()` (or HashMap/HashSet::new()), check `current_function_return_type`. If it's `Vec<T>`, emit `Vec::<T>::new()`.

**Files:** `windjammer/src/codegen/rust/expression_generation.rs`

**Impact:** Fixes svo.wj and similar patterns: `pub fn encode() -> Vec<u32> { return Vec::new() }`

## Patterns Now Handled

| Pattern | Inference Source | Status |
|---------|------------------|--------|
| `let mut x = Vec::new(); x.push(item)` | First .push() in body | ✅ (was broken - MethodCall) |
| `let mut x = Vec::new(); ...; return x` | Function return type | ✅ (was broken - MethodCall) |
| `for ach in map.values() { result.push(ach) }` | Loop var from .values() + push | ✅ (loop_var_types_override) |
| `return Vec::new()` when fn -> Vec<T> | Function return type | ✅ (new turbofish) |
| `let mut x = Vec::new(); while ... { x.push(...) }` | Push in loop body | ✅ (extract_push recurses) |

## Philosophy

**"Compiler Does the Hard Work"** - exploit every bit of context:
- Parser produces MethodCall → handle it
- Variable returned → use fn return type
- Push in loop → recurse into loop body
- Loop over .values() → propagate &V to loop var
